CHAPTER 4

ANALYZING THE PITFALLS, IDENTIFYING THE DEPENDENCIES, AND APPLYING NORMALIZATIONS

(The relations below are prepared for the SecureMeal Hostel Mess Management System project and aligned with the schema used in Units 1, 2, and 3.)


4.1 Analyse the Pitfalls in Relations

Unnormalized Relation (Before Normalization):
MESS_OPERATION_DETAILS(
  student_id, student_name, dept, room_no, phone,
  plan_type, menu_day, meal_type, food_items,
  attendance_date, attendance_status,
  feedback_rating, feedback_comments,
  staff_name, staff_role
)

Before Normalization Output:

+------------+---------------+------+---------+-----------+---------------------------+-------------------------------+-----------------------------+----------------+
| student_id | student_name  | dept | room_no | plan_type | meal_type_values          | attendance_values             | feedback_values             | staff_name     |
+------------+---------------+------+---------+-----------+---------------------------+-------------------------------+-----------------------------+----------------+
| 101        | Aarav Sharma  | CSE  | A-101   | regular   | breakfast,lunch,dinner   | 2026-04-15-present,absent     | 5-Good,4-Needs more salad   | Mess Manager   |
| 102        | Diya Patel    | ECE  | B-204   | veg       | breakfast,lunch          | 2026-04-15-present,present    | 4-Good breakfast            | Head Cook      |
+------------+---------------+------+---------+-----------+---------------------------+-------------------------------+-----------------------------+----------------+


Pitfall Summary Table:

+-------------------+----------------------------------------------------------------------------------+
| Pitfall           | Description                                                                      |
+-------------------+----------------------------------------------------------------------------------+
| Redundancy        | Student, plan, menu, attendance, and staff details repeat in multiple rows.      |
| Update Anomaly    | Changing room_no, food_items, or plan_type requires updates in many tuples.      |
| Insertion Anomaly | New student/menu/staff cannot be inserted independently without unrelated data.   |
| Deletion Anomaly  | Deleting one attendance/feedback row may remove important master data.            |
| Complexity        | One large relation makes querying, updating, and reporting difficult.             |
+-------------------+----------------------------------------------------------------------------------+

CHAPTER 5

IMPLEMENTATION OF CONCURRENCY CONTROL AND RECOVERY MECHANISMS


5.1 ACID Properties
ACID Summary Table:

+------------+--------------------------------------------------------------------------------+
| Property   | Meaning                                                                        |
+------------+--------------------------------------------------------------------------------+
| Atomicity  | All operations in one transaction succeed together, or none are applied.       |
| Consistency| Database moves from one valid state to another valid state.                    |
| Isolation  | Concurrent transactions do not interfere with each other.                      |
| Durability | Committed changes remain permanent even after failure or restart.              |
+------------+--------------------------------------------------------------------------------+


5.2 TCL (Commit, Rollback, Savepoint)
TCL Summary Table:

+------------+----------------------------------------------------------------+
| Command    | Purpose                                                        |
+------------+----------------------------------------------------------------+
| COMMIT     | Permanently saves all successful changes.                      |
| ROLLBACK   | Cancels uncommitted changes and restores previous state.       |
| SAVEPOINT  | Creates an intermediate point for partial rollback.            |
+------------+----------------------------------------------------------------+


5.3 Minimum 5 Transactions


5.3.1 Insert Transaction
mysql> START TRANSACTION;
Query OK, 0 rows affected (0.00 sec)

mysql> INSERT INTO STUDENT (name, dept, room_no, phone)
    -> VALUES ('Neha Singh', 'CSE', 'A-204', '9876501111');
Query OK, 1 row affected (0.01 sec)

mysql> SET @new_student_id = LAST_INSERT_ID();
Query OK, 0 rows affected (0.00 sec)

mysql> INSERT INTO MESS_PLAN (student_id, plan_type)
    -> VALUES (@new_student_id, 'veg');
Query OK, 1 row affected (0.00 sec)

mysql> COMMIT;
Query OK, 0 rows affected (0.01 sec)

Result Table:

+------------+---------------+------+---------+------------+
| student_id | name          | dept | room_no | phone      |
+------------+---------------+------+---------+------------+
| 104        | Neha Singh    | CSE  | A-204   | 9876501111 |
+------------+---------------+------+---------+------------+


5.3.2 Rollback Transaction

+------------------------------+-------+

5.3.3 Savepoint Transaction
mysql> START TRANSACTION;
Query OK, 0 rows affected (0.00 sec)

mysql> INSERT INTO MENU (day, meal_type, food_items)
    -> VALUES ('Tuesday', 'breakfast', 'Idli, Sambar, Tea');
Query OK, 1 row affected (0.01 sec)

mysql> SAVEPOINT breakfast_saved;
Query OK, 0 rows affected (0.00 sec)

mysql> INSERT INTO MENU (day, meal_type, food_items)
    -> VALUES ('Tuesday', 'lunch', 'Rice, Dal, Paneer Curry');
Query OK, 1 row affected (0.00 sec)

mysql> ROLLBACK TO breakfast_saved;
Query OK, 0 rows affected (0.00 sec)

mysql> COMMIT;
Query OK, 0 rows affected (0.01 sec)

Result Table:

+---------+-----------+---------------------+
| day     | meal_type | food_items          |
+---------+-----------+---------------------+
| Tuesday | breakfast | Idli, Sambar, Tea   |
+---------+-----------+---------------------+

5.4 Concurrency Control
Concurrency Control Summary Table:

+-------------------+------------------------------------------------------------------+
| Topic             | Description                                                      |
+-------------------+------------------------------------------------------------------+
| Row-level locking | Locks only the selected rows during transaction execution.       |
| Table-level lock  | Locks the whole table when broad protection is required.         |
| Shared lock       | Read lock; other shared reads may proceed.                      |
| Exclusive lock    | Write lock; conflicting reads/writes are blocked.               |
+-------------------+------------------------------------------------------------------+


Row-level locking example with terminal output:

mysql> START TRANSACTION;
Query OK, 0 rows affected (0.00 sec)

mysql> SELECT *
    -> FROM ATTENDANCE
    -> WHERE student_id = 101
    ->   AND date_val = '2026-04-15'
